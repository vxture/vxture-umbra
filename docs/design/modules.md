## Module 1: Nginx (Gateway)

### Responsibility

- Public SNI stream routing on port 443 (layer 4, no TLS termination for Xray)
- TLS termination and HTTP virtual hosting for all *.ruyin.ai domains
- HTTP port 80: ACME challenge + redirect
- Reverse proxy to all internal services

### Config Structure

Nginx runs two listeners:

**Stream listener (port 443, in `nginx.conf` stream block):**
- `ssl_preread on` - reads SNI without terminating TLS
- Routes to `xray_upstream` (`umbra-marzban:10443`, bundled Xray subprocess) or `https_upstream` (`127.0.0.1:8443`)

**HTTP listener (internal port 8443):**
- TLS termination with Let's Encrypt certs
- One `server {}` block per virtual host

### Virtual Host Configs

| Template | Domain | Upstream |
|----------|--------|---------|
| `00-default.conf.template` | catch-all | ACME challenge and default 404/redirect behavior |
| `01-ruyin.conf.template` | `ruyin.ai` | `umbra-website:3210` Next public website |
| `02-www.conf.template` | `www.ruyin.ai` | 301 canonical redirect to `ruyin.ai` |
| `03-vpn-portal.conf.template` | `EDGE_DOMAIN` | `umbra-website:3210` VPN display and guide |
| `04-sub.conf.template` | `sub.ruyin.ai` | `umbra-subproxy:8080` for native Marzban `/sub/<token>` metadata normalization |
| `05-console.conf.template` | `console.ruyin.ai` | `umbra-account-web:3220` + `umbra-account:3281` |
| `06-pass.conf.template` | `pass.ruyin.ai` | `umbra-vaultwarden:80` |
| `07-admin.conf.template` | `admin.ruyin.ai` | `umbra-marzban:8000` + `/invites` to account web/API |

### Stream Config Spec

```nginx
stream {
    map $ssl_preread_server_name $upstream_backend {
        www.microsoft.com   xray_upstream;
        default             https_upstream;
    }

    upstream xray_upstream {
        server umbra-marzban:10443;
    }

    upstream https_upstream {
        server 127.0.0.1:8443;
    }

    server {
        listen 443;
        proxy_pass     $upstream_backend;
        ssl_preread    on;
        proxy_protocol on;
    }
}
```

### admin.ruyin.ai - Login Boundary

`admin.ruyin.ai` is public at nginx and protected by Marzban's own web login and API JWT session. This allows operators to reach Marzban pages without requiring a separate VPN path first. `console.ruyin.ai` is reserved for the Ruyin user self-service console.

```nginx
server {
    listen 8443 ssl;
    server_name admin.ruyin.ai;

    ssl_certificate /etc/letsencrypt/live/admin.ruyin.ai/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/admin.ruyin.ai/privkey.pem;

    location / {
        proxy_pass https://umbra-marzban:8000;
        proxy_ssl_verify off;
        include /etc/nginx/snippets/proxy-headers.conf;
    }
}
```

#### Console Authentication

Nginx must not add IP allow/deny or `auth_basic` to this vhost. Marzban handles the web login and API JWT session itself; Basic Auth intercepts Bearer-token API calls.

#### Access Flow Summary

```
Browser -> console.ruyin.ai
  Nginx -> umbra-account-web / umbra-account
  Vxture SSO -> invite activation / subscription dashboard

Browser -> admin.ruyin.ai/invites
  Nginx -> umbra-account-web / umbra-account
  Admin -> sign in with Marzban admin account
  Account service -> stores Marzban admin token in a server-side admin session
  Admin -> generate one-time invite for an existing USER**

Browser -> admin.ruyin.ai/dashboard/
  Nginx -> Marzban
  Marzban login -> dashboard / user pages
```

## Module 2: Xray-core

### Responsibility

- VLESS + REALITY + Vision proxy
- Multi-user, UUID-isolated clients
- Managed entirely by Marzban (Marzban writes `xray_config.json` and restarts Xray)

### Xray <-> Marzban Relationship

Marzban manages Xray. The deployment flow is:

1. REALITY keys generated by [`13-generate-runtime-secrets.sh`](../../deploy/worker-03/scripts/13-generate-runtime-secrets.sh)
2. Keys written to `private/reality.json`
3. Marzban configured with REALITY keys at startup
4. Marzban renders its own `xray_config.json` and manages Xray process internally

**Marzban runs Xray as a subprocess** - do NOT run a separate `umbra-xray` container alongside Marzban. Marzban's container includes Xray.

### Xray Config (managed by Marzban)

Key inbound spec that Marzban must be configured with:

```json
{
  "tag": "VLESS_TCP_REALITY",
  "listen": "0.0.0.0",
  "port": 10443,
  "protocol": "vless",
  "settings": {
    "clients": [],
    "decryption": "none"
  },
  "streamSettings": {
    "network": "tcp",
    "security": "reality",
    "realitySettings": {
      "show": false,
      "dest": "www.microsoft.com:443",
      "xver": 0,
      "serverNames": ["www.microsoft.com"],
      "privateKey": "{{ REALITY_PRIVATE_KEY }}",
      "shortIds": ["{{ REALITY_SHORT_ID }}"]
    }
  },
  "sniffing": {
    "enabled": true,
    "destOverride": ["http", "tls", "quic"]
  }
}
```

### Port

| Port | Visibility | Usage |
|------|-----------|-------|
| 10443 | Internal only | VLESS+REALITY inbound |

### Marzban Container Volumes

```yaml
volumes:
  - DATA_DIR/marzban:/var/lib/marzban
```

## Module 3: Marzban

### Responsibility

- VPN user management (create, disable, delete, reset traffic)
- Subscription generation (Clash/Mihomo YAML, V2Ray JSON, etc.)
- Traffic accounting and expiry enforcement
- Xray config management (renders and hot-reloads Xray)
- REST API for automation
- Multi-node support (future: add more edge nodes)

### Image

```
gozargah/marzban:latest
```

### Key Environment Variables

```env
UVICORN_HOST=0.0.0.0
UVICORN_PORT=8000

SQLALCHEMY_DATABASE_URL=sqlite:////var/lib/marzban/db.sqlite3

XRAY_JSON=/var/lib/marzban/xray_config.json
XRAY_EXECUTABLE_PATH=/usr/local/bin/xray
XRAY_ASSETS_PATH=/usr/local/share/xray

SUDO_USERNAME={{ MARZBAN_ADMIN_USER }}
SUDO_PASSWORD={{ MARZBAN_ADMIN_PASSWORD }}

SUBSCRIPTION_URL_PREFIX=https://sub.ruyin.ai

CUSTOM_TEMPLATES_DIRECTORY=/var/lib/marzban/templates
CLASH_SUBSCRIPTION_TEMPLATE=clash/default.yml
```

### Marzban Subscription Template Location

```
DATA_DIR/marzban/templates/clash/default.yml
```

This file is rendered from `configs/marzban/clash-subscription.j2` - it contains the B++ proxy rules. See `decisions.md` and `../implementation/subscriptions.md` for full subscription context.

### Subscription Metadata Normalizer

`umbra-subproxy` is internal-only and sits between nginx and Marzban for
`/sub/<token>`. It does not convert subscription formats. It normalizes the
response filename, `profile-title` header, and YAML `#profile-title` to
`Ruyin-USERNAME`.
This is a narrow edge adapter and can remain Python while it stays small.

### Docker Config

```yaml
umbra-marzban:
  image: gozargah/marzban:latest
  restart: always
  env_file: .env
  volumes:
    - DATA_DIR/marzban:/var/lib/marzban
  networks:
    - umbra-net
```

## Module 4: Storage (SQLite)

All stateful services use SQLite on this 1C1G node; no PostgreSQL container. See
[`decisions.md`](decisions.md) section 3 for the rationale. Per-service files:

| File | Service |
|------|---------|
| `DATA_DIR/marzban/db.sqlite3` | Marzban |
| `DATA_DIR/account/account.db` | Account portal |
| `DATA_DIR/vaultwarden/data/db.sqlite3` | Vaultwarden |

### Backup

SQLite databases are backed up via file copy or root-container archives in `deploy/worker-03/scripts/55-backup-runtime-state.sh`. Vaultwarden's full `data/` directory is archived (includes attachments and sends). Account portal state is archived as `account-data-<timestamp>.tar.gz`.

## Module 5: Account Console (console.ruyin.ai)

### Responsibility

- User-facing account entry point for the VPN service
- Password login for invite-bound users
- Dashboard with user code, copy-only subscription URL, status, quota, used traffic, remaining traffic, expiry, and last online time
- Registration with one-time invite codes generated by an admin
- User-chosen display name for portal login
- Invite codes bind to one existing Marzban user such as `USER08`; users cannot choose a different user code
- The dashboard reads the stored subscription URL on normal page views. The user `Reset subscription URL` button and admin invite-console `Reset URL` button call the same reset path: refresh the current `subscription_url` from Marzban by bound user code, then update the local account cache. Reset buttons are styled as danger actions because they can change the URL users copy into clients.

### Stack

Self-contained Python stdlib HTTP service with SQLite state in `DATA_DIR/account/account.db`.
This is the current lightweight edge-node implementation. New formal Ruyin
business backends should be implemented with NestJS; migrate this account API
when account, invite, billing, audit, or admin workflows exceed the current
single-file service scope.

### Content Structure

```
vpn.ruyin.ai/
`-- /                  -> VPN display via umbra-website (/guide/ redirects to /)

console.ruyin.ai/
|-- /                  -> login / account home
|-- /register          -> invite activation
`-- /dashboard         -> authenticated subscription dashboard

admin.ruyin.ai/invites
|-- /                  -> Marzban admin login and user binding table
|-- /create            -> creates a one-time invite for an unbound Marzban user
`-- /revoke            -> disables an unused invite
```

### Docker Config

```yaml
umbra-account:
  image: python:3.12-alpine
  restart: always
  environment:
    ACCOUNT_DB_PATH: /var/lib/umbra-account/account.db
    ACCOUNT_SESSION_SECRET: ${ACCOUNT_SESSION_SECRET}
    ACCOUNT_INVITE_SECRET: ${ACCOUNT_INVITE_SECRET}
    MARZBAN_ADMIN_USER: ${MARZBAN_ADMIN_USER}
    MARZBAN_ADMIN_PASSWORD: ${MARZBAN_ADMIN_PASSWORD}
  volumes:
    - DATA_DIR/account:/var/lib/umbra-account
    - ./services/account/account.py:/app/account.py:ro
  networks:
    - umbra-net
```

## Module 7: Certbot

### Responsibility

- Issue Let's Encrypt certs for active domains
- Auto-renew via cron

### Issue Command

All domains in a single cert (or separate certs per domain - separate recommended for easier rotation):

```bash
# Issue per domain (recommended)
certbot certonly --webroot \
  --webroot-path DATA_DIR/certbot \
  --email "$CERTBOT_EMAIL" \
  --agree-tos --non-interactive \
  -d vpn.ruyin.ai

# Repeat for each domain
```

Or as a wildcard cert (requires DNS challenge, more complex):

```bash
certbot certonly --dns-cloudflare \
  --dns-cloudflare-credentials /path/to/cf-creds.ini \
  -d "*.ruyin.ai" -d "ruyin.ai"
```

**Recommendation:** Per-domain webroot for simplicity in v1.0. Consider wildcard in v1.1 if managing 9 certs becomes maintenance burden.

### Renewal Script

```bash
bash deploy/worker-03/ops.sh certs --renew
```

Renewal delegates to `certbot renew`; it does not force reissue. Before renewal, scripts remove only invalid zero-byte files under `DATA_DIR/letsencrypt/renewal/`.

Renewal uses certbot's deploy hook marker. If no certificate is renewed, nginx and Marzban are left untouched. If a renewal happens, nginx is config-tested and reloaded, and Marzban is restarted after the edge cert is synced into `DATA_DIR/marzban/tls`.

Real-certificate upgrade must use `bash deploy/worker-03/ops.sh certs --upgrade`. It normalizes legacy workdirs, copies existing certificate state into `DATA_DIR/letsencrypt.staged`, reuses trusted LE certs that are not near expiry, issues only missing/expiring/non-trusted domains in staging, keeps partial staged successes after a failed run, verifies all staged domains independently for issuer, expiry, and name match, and activates the staged directory only after every domain succeeds.

### Cron

```cron
17 3 * * * /srv/vxture/repo/umbra/deploy/worker-03/ops.sh certs --renew >> /var/log/umbra-cert-renew.log 2>&1
```
